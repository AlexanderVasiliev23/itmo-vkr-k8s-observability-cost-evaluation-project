# Модель оценки ресурсов

## Назначение модели

Экспериментальные прогоны дают измерения только в отдельных точках: конкретный инструмент, конкретная нагрузка, конкретная длительность прогона. Практическая задача ObsAdvisor — оценить потребление ресурсов при пользовательских параметрах, которые могут не совпадать с измеренными точками.

Модель строится на лету при каждом запросе на основе актуальных данных из SQLite.

## Используемые ресурсы

В модели используются три ресурса:

- `cpu_cores` — среднее потребление CPU;
- `mem_peak_bytes` — пиковое потребление RAM;
- `disk_bytes` — максимальное дисковое потребление PVC.

## Интерполяция по нагрузке

Для всех трёх ресурсов используется кусочно-линейная интерполяция по `load_value`.

Если целевая нагрузка находится между измеренными точками, значение рассчитывается линейной интерполяцией между ближайшими точками.

Если целевая нагрузка находится за пределами измеренного диапазона, используется линейная экстраполяция по ближайшему крайнему сегменту. Такие оценки следует интерпретировать осторожно.

## Обработка повторных точек

Если в базе есть несколько прогонов с одинаковым `load_value`, они схлопываются через медиану. Это снижает влияние отдельных шумных измерений и повторных запусков.

## Масштабирование дискового потребления

CPU и RAM в текущей модели не масштабируются по retention. Они оцениваются по нагрузке и рассматриваются как ресурсы, зависящие преимущественно от текущей интенсивности обработки данных.

Для диска дополнительно учитывается горизонт хранения данных. Используется степенное масштабирование:

```text
D(T) = D_base × (T / T_base)^α
```

где:

- `D(T)` — оценка дискового потребления при целевом горизонте хранения;
- `D_base` — измеренное или оцененное дисковое потребление в базовой точке;
- `T_base` — фактическая длительность накопления данных в экспериментальном прогоне;
- `T` — целевой горизонт хранения;
- `α` — показатель масштабирования диска по времени.

## Оценка α

Показатель `α` оценивается по калибровочным парам: строки с одинаковым инструментом и одинаковой нагрузкой, но разной длительностью прогона.

Используется логарифмическая регрессия зависимости `log(disk)` от `log(duration)` внутри групп с одинаковым `load_value`. Если доступно несколько групп, итоговое значение берётся как медиана оценённых `α`.

Если калибровочных пар нет, используется значение:

```text
α = 1.0
```

Оно соответствует линейному росту дискового потребления во времени.

Для защиты от шумных измерений `α` ограничивается диапазоном:

```text
0.6 ≤ α ≤ 1.4
```

## Калибровочные 24h-прогоны

Калибровочные 24h-прогоны используются для уточнения дисковой модели. Они включаются в модель дискового потребления, но не используются как базовые точки для CPU/RAM.

Это сделано потому, что длительные прогоны предназначены для оценки роста накопительного ресурса — диска, а не для смешивания базовых точек по нагрузке с калибровочными измерениями.

## Проверка качества модели

Для диагностики качества используется leave-one-out cross-validation.

Калибровочные 24h-прогоны не используются как тестовые точки. Проверка выполняется по базовым прогонам:

- для CPU и RAM проверяется качество интерполяции по нагрузке;
- для диска проверяется оценка с учётом фактического периода накопления данных.

Метрика ошибки:

```text
MAPE = mean(|y_true - y_pred| / y_true)
```

Порог 20% используется как инженерный ориентир качества модели, а не как строгий статистический критерий.

## Диапазон неопределённости

Кроме точечной оценки, ObsAdvisor возвращает диапазон с учётом error budget:

```text
min = estimate × (1 - error_budget)
max = estimate × (1 + error_budget)
```

Если пользователь не задаёт error budget явно, используется значение по умолчанию 20% (`error_budget = 0.2`). Этот диапазон отражает неопределённость модели и вариативность измерений.
