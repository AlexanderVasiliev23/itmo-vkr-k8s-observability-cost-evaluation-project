# Ограничения

## Конфигурация стенда

Результаты относятся к конкретной Kubernetes-инфраструктуре, параметрам узлов, версиям инструментов и настройкам развёртывания. Их следует интерпретировать как сравнительные измерения в выбранной конфигурации, а не как универсальные абсолютные характеристики инструментов.

## Нагрузка на запись

Основной фокус экспериментов — ресурсное потребление при заданной интенсивности записи данных:

- для метрик — число активных временных рядов;
- для логов — число строк в секунду.

Максимальная пропускная способность записи не исследуется.

## Нагрузка на чтение

Фоновая нагрузка на чтение фиксирована на уровне 1 QPS и не моделирует полноценный read path. Latency, throughput и сложные пользовательские сценарии поиска не являются предметом текущей работы.

## Синтетическая нагрузка

Для метрик используется контролируемый генератор временных рядов, а для логов — генератор строк фиксированного размера. Такая нагрузка обеспечивает воспроизводимость, но не охватывает всё разнообразие production-сценариев.

## Топология scrape-targets

Для метрической нагрузки может использоваться один endpoint с заданным числом временных рядов. В production-средах та же кардинальность может распределяться между сотнями или тысячами целей, что может изменить профиль потребления CPU и RAM.

## Модель оценки

Модель предназначена для инженерной оценки. Экстраполяцию за пределы измеренного диапазона нагрузок следует интерпретировать осторожно.

CPU и RAM в текущей модели не масштабируются по retention. Для длительных горизонтов хранения это допущение может быть недостаточным для инструментов, у которых фоновые процессы зависят от накопленного объёма данных.

## Стоимость

Стоимость рассчитывается по CPU, RAM и диску. Она не учитывает сетевой трафик, snapshot, IOPS, стоимость managed-сервисов, лицензирование, трудозатраты на эксплуатацию и другие компоненты полного TCO.
