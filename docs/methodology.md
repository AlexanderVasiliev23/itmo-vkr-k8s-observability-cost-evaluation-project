# Методика экспериментов

## Общая идея

Цель экспериментов — получить сопоставимые измерения потребления CPU, RAM и дискового пространства для выбранных observability-инструментов в контролируемой Kubernetes-среде.

Варьируется основной параметр нагрузки:

- для систем метрик — число активных временных рядов;
- для систем логирования — число логовых строк в секунду.

Остальные параметры фиксируются, чтобы отделить влияние архитектуры инструмента от влияния инфраструктуры и методики измерения.

## Фиксированные условия

| Параметр | Значение | Назначение |
|----------|----------|------------|
| Интервал сбора метрик | 15 с | Типовое значение для Prometheus-окружений |
| Число лейблов на серию | 1 (`series_id`) | Изоляция эффекта кардинальности |
| Размер строки лога | фиксированный | Изоляция эффекта интенсивности записи |
| Retention | 7 дней | Базовый горизонт для сравнительных прогонов |
| Фоновая нагрузка на чтение | 1 QPS | Исключение полностью idle-сценария |
| Прогрев | 15 минут | Исключается из окна измерений |

Допустимые значения `retention_days` в CLI: `1`, `7`, `30`.

## Типы прогонов

| Тип прогона | Длительность | Назначение |
|-------------|--------------|------------|
| Базовый | 3h | Основная серия измерений по оси нагрузки |
| Укороченный | 1.5h | Используется для части логовых сценариев |
| Калибровочный | 24h | Оценка роста дискового потребления во времени |

Калибровочные прогоны используются прежде всего для оценки показателя `α` в модели дискового потребления.

## Период прогрева и окно измерений

Первые 15 минут прогона исключаются из расчёта. Основное окно измерений рассчитывается как:

```text
T_collect = duration - 15 минут
```

Например:

- для 3h-прогона `T_collect = 165 минут`;
- для 24h-прогона `T_collect = 1425 минут`.

## Сбор метрик

ObsBench не собирает метрики напрямую в ходе эксперимента. В ходе прогона служебный Prometheus накапливает данные от cAdvisor и `disk-usage-exporter`. После завершения прогона ObsBench выполняет PromQL-запросы к служебному Prometheus и получает агрегированные значения.

## Измеряемые ресурсы

### CPU

CPU оценивается как среднее потребление процессора за основное окно измерений. Источник данных — `container_cpu_usage_seconds_total` через cAdvisor.

### RAM

Для модели используется пиковое значение `container_memory_working_set_bytes`. Пиковая RAM важна для оценки требуемого лимита памяти и риска OOMKill.

### Диск

Дисковое потребление измеряется через `disk-usage-exporter`, который монтирует PVC тестируемого инструмента и экспортирует объём занятого пространства в байтах. Для результата используется максимум за основное окно измерений.

## Нагрузка на чтение

Во время экспериментов поддерживается фиксированная фоновая нагрузка на чтение — 1 QPS. Она не предназначена для полноценного исследования read path, а используется для исключения полностью idle-сценария.

Производительность запросов на чтение, включая latency, throughput и сложные аналитические сценарии, выходит за рамки текущей работы.
