# Сравнительный анализ инструментов наблюдаемости в Kubernetes

Проект содержит инструментарий для воспроизводимого сравнения ресурсного потребления observability-инструментов в Kubernetes-кластере и оценки ориентировочной стоимости их эксплуатации.

Поддерживаются две группы инструментов:

- **Метрики:** Prometheus, VictoriaMetrics
- **Логи:** Grafana Loki, OpenSearch

В проект входят два основных компонента:

- **ObsBench** — CLI-инструмент для запуска нагрузочных экспериментов, развёртывания тестируемых инструментов, генерации нагрузки и сбора метрик CPU, RAM и дискового пространства.
- **ObsAdvisor** — веб-приложение для анализа накопленных экспериментальных данных, оценки ресурсов и расчёта ориентировочной стоимости эксплуатации.

## Быстрый старт

### Запуск ObsAdvisor

```bash
cd obs-advisor
make run
```

После запуска интерфейс доступен по адресу:

```text
http://localhost:8086/
```

`make run` запускает ObsAdvisor с базой `../data/obs_bench_results.db`. Для другой базы используйте обязательный флаг `--db`:

```bash
./server/obs-advisor --db /path/to/obs_bench_results.db --static .
```

### Запуск одного эксперимента ObsBench

```bash
cd obs-bench
make build

./obs-bench run-experiment \
  --instrument prometheus \
  --load-value 50000 \
  --retention-days 7 \
  --duration 3h
```

Допустимые значения `--retention-days`: `1`, `7`, `30`.

### Запуск серии экспериментов

```bash
./obs-bench run-batch --file experiments-final.yaml
```

При ошибке отдельного эксперимента batch-запуск продолжает выполнение остальных прогонов, а по завершении выводит итоговый статус.

## Архитектура проекта

Общий поток данных:

```text
ObsBench
  ├── разворачивает инструмент в Kubernetes
  ├── запускает генератор нагрузки
  ├── получает агрегированные метрики через служебный Prometheus
  └── сохраняет результат в SQLite
             │
             ▼
data/obs_bench_results.db
             │
             ▼
ObsAdvisor
  ├── читает экспериментальные точки
  ├── строит модель оценки ресурсов
  ├── рассчитывает ориентировочную стоимость
  └── отображает результаты в веб-интерфейсе
```

ObsBench и ObsAdvisor разделены намеренно: сбор экспериментальных данных и последующий анализ являются разными задачами. ObsBench рассчитан на длительные прогоны в облачной инфраструктуре, а ObsAdvisor может запускаться локально для анализа уже собранной SQLite-базы.

## Методика экспериментов

В экспериментах фиксируются параметры стенда, длительность прогонов, период прогрева, retention и фоновая нагрузка на чтение. Варьируется основной параметр нагрузки:

- для систем метрик — число активных временных рядов;
- для систем логирования — число логовых строк в секунду.

Первые 15 минут каждого прогона считаются периодом прогрева и исключаются из расчёта. После завершения прогона ObsBench обращается к служебному Prometheus и получает агрегированные показатели за основное окно измерений.

Измеряются:

- **CPU** — среднее потребление процессора за основное окно измерений;
- **RAM** — пиковое значение `container_memory_working_set_bytes`;
- **Disk** — максимальный объём занятого PVC через `disk-usage-exporter`.

Подробнее: [Методика экспериментов](docs/methodology.md).

## Модель оценки ресурсов

ObsAdvisor строит модель на лету при каждом пользовательском запросе на основе актуальных строк из SQLite.

Модель включает два механизма:

1. **Интерполяция по нагрузке.**  
   CPU, RAM и диск оцениваются по экспериментальным точкам с разными значениями `load_value`.

2. **Масштабирование дискового потребления по горизонту хранения.**  
   Для `disk_bytes` дополнительно применяется степенное масштабирование:

   ```text
   D(T) = D_base × (T / T_base)^α
   ```

CPU и RAM не масштабируются по retention в текущей модели: они оцениваются как ресурсы, зависящие преимущественно от текущей интенсивности обработки данных.

Подробнее: [Модель оценки ресурсов](docs/capacity-model.md).

## Расчёт стоимости

Ориентировочная стоимость эксплуатации рассчитывается линейно по трём компонентам:

```text
Cost = CPU_cores × price_cpu + RAM_GiB × price_ram + Disk_GiB × price_disk
```

Тарифы задаются пользователем в интерфейсе ObsAdvisor. Значения по умолчанию могут соответствовать выбранному облачному провайдеру, но модель не привязана к конкретной платформе.

Подробнее: [Расчёт стоимости](docs/pricing.md).

## Ограничения

Результаты следует интерпретировать как инженерную оценку в пределах выбранной конфигурации стенда. Основные ограничения:

- сравнение проводится в фиксированной Kubernetes-инфраструктуре;
- нагрузка на чтение зафиксирована и не моделирует полноценный read path;
- для метрик используется контролируемая синтетическая нагрузка;
- стоимость является нормализованной оценкой по CPU, RAM и диску, а не полным облачным биллингом;
- экстраполяция за пределы измеренного диапазона требует осторожной интерпретации;

Подробнее: [Ограничения](docs/limitations.md).

## Запуск на runner VM

Длительные прогоны удобно выполнять на отдельной виртуальной машине в той же облачной инфраструктуре, где развёрнут Kubernetes-кластер. Это снижает риск прерывания эксперимента из-за нестабильного локального окружения.

Подробнее: [Запуск экспериментов на runner VM](docs/runner-vm.md).
